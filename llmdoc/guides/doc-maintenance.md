# 文档维护指南

## 何时必须更新 llmdoc

出现以下任一情况时，应同步检查并更新 `llmdoc/`：

- 新增或重命名核心模块、路由、页面、脚本
- 写入链路、检索链路、治理链路发生行为变化
- 部署方式、环境变量、默认端口或鉴权模式发生变化
- 文档中“建议先看哪里”的入口已经失效
- 对外说明边界变化，导致原本的项目认知顺序不再成立

## 维护原则

- 只写已确认的事实
  - 以当前仓库代码、测试和现有 `docs/` 为准
- 保持最小
  - `llmdoc/` 负责导航和决策支持，不复制整份公开文档
- 优先可检索
  - 文档标题、文件名、关键术语要能被快速 grep 到
- 优先稳定信息
  - 多写架构入口、职责边界、阅读顺序
  - 少写容易频繁漂移的数字和一次性结论

## 推荐更新方法

1. 先确认变更影响的是：
   - overview
   - architecture
   - guides
   - reference
2. 只修改受影响的最小文档
3. 若已有公开文档更详细，`llmdoc/` 只补导航和摘要，不重复展开
4. 修改后至少做一次路径自检，确认引用文件仍存在

## Windows 终端提示

如果在 PowerShell 下读取中文 markdown 出现乱码，优先：

1. 切 UTF-8 输出
2. 使用 `Get-Content -Encoding utf8`

否则很容易把现有文档读错，再把错误理解写回 `llmdoc/`。

## 与提交/PR 的同步规则

- 代码行为变化且影响理解路径时，代码与 `llmdoc` 应尽量同一批提交
- 如果本次只修实现、不改变认知模型，可不改 `llmdoc`
- 如果 README / `docs/` 已更新但 `llmdoc/` 仍指向旧入口，应一并修正
- 对“尚未验证”的内容，不要写成确定事实；应明确标注为待验证

## 推荐校验动作

- 检查 `llmdoc/index.md` 的阅读顺序是否仍成立
- 检查 `architecture/retrieval-map.md` 指向的关键入口是否仍存在
- 检查 `reference/coding-conventions.md` 是否仍符合仓库当前风格
- 必要时补充一条“这次为什么需要更新文档”的提交说明

## 不建议的做法

- 把 `llmdoc/` 写成第二套 README
- 复制大量实现细节，导致文档比代码更快过时
- 用推测替代事实
- 修改代码后只更 README，不修 `llmdoc` 中已失效的导航
