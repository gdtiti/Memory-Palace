# 编码约定

## 命名约定

### Python / 后端

- 模块、函数、变量使用 `snake_case`
- 类名使用 `PascalCase`
- 环境变量使用全大写下划线风格，如 `DATABASE_URL`
- 路由按职责分组，当前主分组为：
  - `browse`
  - `review`
  - `maintenance`

### React / 前端

- 组件文件与组件名使用 `PascalCase`
- hooks、工具函数与普通变量使用 `camelCase`
- `features/` 按页面或能力分目录，不把跨页面通用逻辑塞进页面组件

### 领域语义

- “记忆路径”优先保留 `domain://path` 语义
- 检索模式使用明确字面值：`keyword`、`semantic`、`hybrid`
- Write Guard 动作使用稳定枚举：`ADD`、`UPDATE`、`NOOP`、`DELETE`

## 错误处理约定

### 后端

- 启动期错误应尽早失败，不做静默吞错
- 对外接口优先返回结构化结果，而不是模糊字符串
- 若存在降级路径，应显式暴露降级信息，例如 `degrade_reasons`
- 鉴权相关问题要清楚区分：
  - 未配置
  - 配置错误
  - 权限不足

### 前端

- 页面错误优先展示可操作信息，不只打印到控制台
- 鉴权状态应与运行时注入逻辑保持一致
- 不把后端契约变化悄悄吞进 UI 兼容层里

## 测试与验证约定

### 后端

- 优先运行与改动最接近的 `pytest` 用例
- 涉及写入、检索、鉴权、回滚等契约时，优先补或改对应测试
- 若是运行时或部署问题，至少验证 `/health` 和关键入口是否可达

### 前端

- 优先运行相关 `vitest` 测试
- 页面改动至少检查对应 feature 测试与 API 客户端测试

### 通用

- 先做最小验证，再扩大验证范围
- 不把未验证结论写成“已确认”
- 若因环境限制无法跑完整验证，应明确写出：
  - 已验证部分
  - 未验证部分
  - 剩余风险

## 文档与代码一致性

- 影响架构入口、关键链路、部署方式时，要同步检查 `llmdoc/`
- 文档以“帮助快速定位正确代码”为目标，不追求复制所有实现
